## Prerequisites

Before deploying, make sure you have completed the following:

- A Supabase project with all tables, the `users_view` view, the `mp4` storage bucket, RLS policies, and Google OAuth configured. See [Supabase setup](/self-hosting/supabase-setup).
- The three required environment variables ready. See [Environment variables](/self-hosting/environment-variables).
- Node.js 18 or later (or Bun) if you are running Snippet locally.

## Deploy to Vercel (recommended)

Vercel is the recommended hosting platform for Snippet. It supports the Next.js App Router, server actions, and edge middleware without any additional configuration.

<Steps>
  <Step title="Fork or clone the repository">
    Fork the Snippet repository to your own GitHub account so Vercel can access it, or push the code to any Git provider that Vercel supports (GitHub, GitLab, Bitbucket).

    ```bash
    git clone https://github.com/ehcaw/snippet.git
    cd snippet
    ```
  </Step>

  <Step title="Create a new Vercel project">
    Go to [vercel.com](https://vercel.com) and click **Add New → Project**. Import the repository you just forked or pushed. Vercel automatically detects the Next.js framework and sets the correct build settings.
  </Step>

  <Step title="Add environment variables">
    Before clicking **Deploy**, expand the **Environment Variables** section and add all three variables:

    | Variable | Value source |
    |---|---|
    | `NEXT_PUBLIC_SUPABASE_URL` | Supabase → Project Settings → API → Project URL |
    | `NEXT_PUBLIC_SUPABASE_KEY` | Supabase → Project Settings → API → anon / public |
    | `SUPABASE_SERVICE_ROLE_KEY` | Supabase → Project Settings → API → service_role |

    <Warning>
      Set the environment for `SUPABASE_SERVICE_ROLE_KEY` to **Production** and **Preview** only. Do not select **Development** unless you are certain your local environment does not expose it.
    </Warning>
  </Step>

  <Step title="Deploy">
    Click **Deploy**. Vercel runs `next build` and publishes the application. The first build typically takes two to three minutes.

    Once deployed, Vercel assigns a URL in the format `https://your-project-name.vercel.app`. Copy this URL and add it to the Supabase **Authentication → URL Configuration** as described in the [Supabase setup guide](/self-hosting/supabase-setup#enable-google-oauth).
  </Step>

  <Step title="Set your custom domain (optional)">
    In the Vercel project dashboard, go to **Settings → Domains** and add your custom domain. After DNS propagation completes, update the Supabase **Site URL** and **Redirect URLs** to use your custom domain.
  </Step>
</Steps>

## Run locally for development

<Steps>
  <Step title="Install dependencies">
    From the repository root, install the project's dependencies:

    <CodeGroup>

    ```bash npm
    npm install
    ```

    ```bash yarn
    yarn install
    ```

    ```bash pnpm
    pnpm install
    ```

    ```bash bun
    bun install
    ```

    </CodeGroup>
  </Step>

  <Step title="Create the environment file">
    Copy the three environment variables into a `.env.local` file at the repository root:

    ```bash
    # .env.local
    NEXT_PUBLIC_SUPABASE_URL=https://<your-project-ref>.supabase.co
    NEXT_PUBLIC_SUPABASE_KEY=<your-anon-key>
    SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>
    ```
  </Step>

  <Step title="Start the development server">
    <CodeGroup>

    ```bash npm
    npm run dev
    ```

    ```bash yarn
    yarn dev
    ```

    ```bash pnpm
    pnpm dev
    ```

    ```bash bun
    bun dev
    ```

    </CodeGroup>

    Open [http://localhost:3000](http://localhost:3000) in your browser. Next.js hot-reloads the application as you edit files.

    <Tip>
      Make sure `http://localhost:3000/auth/callback` is listed in the **Redirect URLs** in your Supabase **Authentication → URL Configuration** settings. Without it, Google OAuth redirects will fail during local development.
    </Tip>
  </Step>
</Steps>

## Build for production (self-managed server)

If you are deploying to your own server rather than Vercel, build the application and start the Node.js server:

<Steps>
  <Step title="Build the application">
    <CodeGroup>

    ```bash npm
    npm run build
    ```

    ```bash yarn
    yarn build
    ```

    ```bash pnpm
    pnpm build
    ```

    ```bash bun
    bun run build
    ```

    </CodeGroup>

    Next.js compiles the application into the `.next` directory. Build errors (type errors, missing environment variables, etc.) surface here.
  </Step>

  <Step title="Start the production server">
    <CodeGroup>

    ```bash npm
    npm start
    ```

    ```bash yarn
    yarn start
    ```

    ```bash pnpm
    pnpm start
    ```

    ```bash bun
    bun start
    ```

    </CodeGroup>

    The server listens on port 3000 by default. Use a reverse proxy (Nginx, Caddy, etc.) to serve it on port 443 with TLS.
  </Step>
</Steps>

<Note>
  The `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_KEY` variables are inlined into the client bundle at **build time**. If you change these values after building, you must rebuild the application.
</Note>

## How the auth middleware works

Snippet registers a Next.js middleware in `middleware.ts` that runs on every request except static assets and image files. The middleware calls `updateSession`, which:

1. Creates a Supabase server client using the incoming request cookies.
2. Calls `supabase.auth.getUser()` to validate and refresh the session token.
3. If no valid session exists and the request path is not `/login` or `/auth/*`, redirects the user to `/login`.
4. Returns the response with updated session cookies set, so the browser and server stay in sync.

This means every page in the application except `/login` and the OAuth callback route at `/auth/callback` requires an authenticated session. Users who are not signed in are redirected to `/login` automatically.

```typescript
// middleware.ts
export const config = {
  matcher: [
    "/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)",
  ],
};
```

The matcher excludes Next.js internal paths and static image files so that the middleware only runs on application routes, keeping cold-start latency low.
